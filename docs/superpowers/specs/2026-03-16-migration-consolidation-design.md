# Migration Consolidation: Single-Track Manual Migrations

**Date:** 2026-03-16
**Status:** Draft
**Scope:** Database migration system — consolidate Drizzle + manual migrations into one hand-written SQL track

## Problem

Breeze has two parallel migration systems that depend on each other but run in the wrong order:

1. **Drizzle migrations** (5 files in `apps/api/drizzle/`) — generated by `drizzle-kit`, run first
2. **Manual SQL migrations** (64 date-prefixed files + 1 optional TimescaleDB file in `apps/api/src/db/migrations/`) — hand-written, run second

This causes recurring production issues:
- Drizzle generates invalid PostgreSQL (`CREATE TYPE IF NOT EXISTS` — not valid syntax)
- Drizzle migration 0003 references `search_vector`, a column that only exists after manual migrations run (circular dependency)
- Drizzle migrations lack `IF EXISTS` guards on DROP statements, breaking upgrades from different schema histories
- A fixer script (`fix-drizzle-enums.sh`) is required to patch Drizzle's output — fragile and error-prone
- Migration 0003 has been patched three times post-ship, changing its hash each time

Drizzle ORM remains excellent for type-safe queries. The migration *generator* is the problem.

## Decision

**Stop using `drizzle-kit` for migration generation.** Consolidate into a single track of hand-written, numbered SQL migrations. Keep Drizzle ORM for queries and `drizzle-kit` for schema drift detection only.

## Design

### 1. File Structure

```
apps/api/migrations/
  0001-baseline.sql                        # Full current schema — idempotent
  0002-add-gatekeeper-enabled.sql          # ← from 2026-02-07-add-gatekeeper-enabled.sql
  0003-policy-state-telemetry.sql
  ...
  0065-users-setup-completed-at.sql        # ← from 2026-03-13-users-setup-completed-at.sql
  0066-<next-new-migration>.sql            # First migration after consolidation
```

**64 date-prefixed manual migrations** become `0002` through `0065`. The first post-consolidation migration is `0066`.

**`timescaledb-setup.sql`** is excluded from the numbered sequence. It requires the TimescaleDB extension which is not present on all deployments. It stays as a standalone optional script at `apps/api/migrations/optional/timescaledb-setup.sql` with a guard at the top (`CREATE EXTENSION IF NOT EXISTS timescaledb`). The runner does not auto-apply files in `optional/` — they must be run manually by deployers who want TimescaleDB.

**Naming:** `NNNN-<descriptive-slug>.sql` — zero-padded 4-digit sequence, applied in lexicographic order.

**Rules:**
- Every migration must be fully idempotent: `IF NOT EXISTS`, `IF EXISTS`, `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object` blocks
- Never edit a shipped migration — fix forward with a new migration
- `0001-baseline.sql` is special: creates the entire schema from scratch, skipped on existing databases

**What moves where:**
- `apps/api/drizzle/*.sql` — deleted (content folded into `0001-baseline.sql`)
- `apps/api/drizzle/meta/` — deleted
- `apps/api/src/db/migrations/*.sql` (64 date-prefixed files) — renamed with sequential numbering, moved to `apps/api/migrations/`
- `apps/api/src/db/migrations/timescaledb-setup.sql` — moved to `apps/api/migrations/optional/`
- `apps/api/src/db/migrations/README.md` — TimescaleDB documentation preserved in `apps/api/migrations/optional/README.md`

### 2. Migration Runner

Rewrite `autoMigrate.ts` with a simplified single-track flow using the `postgres` package (same driver already used by the current `autoMigrate.ts`):

```
1. Connect to database (single connection via `postgres` package, max: 1)
2. Ensure tracking table exists:
   CREATE TABLE IF NOT EXISTS breeze_migrations (
     filename TEXT PRIMARY KEY,
     checksum TEXT NOT NULL,
     applied_at TIMESTAMPTZ DEFAULT now()
   )
3. Detect database state:
   a) Fresh database (no `users` table)
      → Run all migrations starting from 0001-baseline.sql
   b) Legacy database (has `users` table AND no `breeze_migrations` table)
      → Mark migrations 0001 through 0065 as applied (these represent current schema)
   c) Normal startup (has `breeze_migrations` table)
      → Run only pending migrations
4. Auto-seed if no users exist (via imported `seed()` which uses the global Drizzle `db` instance —
   the global Drizzle client must be initialized before this step, same as current code)
```

**Legacy detection (step 3b):** The condition is simply `users` table exists AND `breeze_migrations` table does not exist. No need to check for `drizzle.__drizzle_migrations` or `manual_sql_migrations` — the presence of `users` alone proves the database has data, and the absence of `breeze_migrations` proves it hasn't been migrated to the new system. This handles all legacy states: Drizzle-migrated, manually-migrated, `db:push`-created, or any combination.

**Behaviors:**
- Migrations applied in filename order (lexicographic sort on `NNNN-` prefix)
- Each migration runs in its own transaction (PostgreSQL DDL is transactional)
- Checksum validated before applying — if a tracked file's SHA-256 changed, abort with error
- On failure: transaction rolls back, API exits with clear error message, no partial state

**What gets removed from `autoMigrate.ts`:**
- `migrate()` call (Drizzle migrator)
- `baselineMigrations()` function
- `schemaExists()` check for `drizzle` schema
- Import of `drizzle-orm/postgres-js/migrator`
- Import of `runManualSqlMigrations` from `./migrations/run`

**Other files removed/replaced:**
- `apps/api/src/db/migrations/run.ts` — logic absorbed into the new runner
- `apps/api/src/db/migrate.ts` — standalone migration script that also calls `runManualSqlMigrations()` and spawns `drizzle-kit migrate`. Must be deleted or rewritten to call the new runner.

**Legacy table cleanup:** The old `manual_sql_migrations` and `drizzle.__drizzle_migrations` tables are left in place on existing databases (harmless). A future migration (`0066+`) can drop them if desired, but it's not required for correctness.

### 3. Baseline Generation (`0001-baseline.sql`)

**Process:**
1. Start a fresh Postgres container
2. Run current `autoMigrate()` against it (both Drizzle + manual migrations)
3. Dump schema: `pg_dump --schema-only --no-owner --no-privileges`
4. Make idempotent: wrap `CREATE TYPE` in `DO $$ BEGIN ... EXCEPTION` blocks, add `IF NOT EXISTS` to `CREATE TABLE`/`CREATE INDEX`
5. Validate: run against a second fresh Postgres, `pg_dump` both, diff must be identical

**File structure:**
```sql
-- 0001-baseline.sql
-- Full Breeze schema baseline — generated from Drizzle migrations + manual migrations
-- This file is idempotent and safe to re-run.

-- Extensions
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- Enums
DO $$ BEGIN CREATE TYPE "public"."device_status" AS ENUM(...); EXCEPTION WHEN duplicate_object THEN null; END $$;
...

-- Tables
CREATE TABLE IF NOT EXISTS "users" (...);
CREATE TABLE IF NOT EXISTS "devices" (...);
...

-- Generated columns
ALTER TABLE "device_event_logs" ADD COLUMN IF NOT EXISTS "search_vector" tsvector
  GENERATED ALWAYS AS (...) STORED;

-- Indexes
CREATE INDEX IF NOT EXISTS ...;

-- Foreign keys

-- RLS functions
CREATE OR REPLACE FUNCTION breeze_current_scope() ...;

-- RLS policies
```

### 4. Drizzle ORM Integration

**Stays:**
- All Drizzle schema definitions in `apps/api/src/db/schema/` (drive type-safe queries)
- `drizzle()` instance for query building
- All route/service code using Drizzle for DML (SELECT/INSERT/UPDATE/DELETE)
- `db:studio` script (Drizzle Studio reads schema definitions, not migration files)

**Changes:**
- `drizzle-kit` becomes dev dependency only
- `db:generate` script replaced with `db:check-drift`:
  - Runs `drizzle-kit generate` to a temp folder (not `apps/api/drizzle/`)
  - Reports "schema drift detected" if anything was produced
  - Cleans up — never writes real migrations
- `db:push` removed
- `fix-drizzle-enums.sh` removed
- `drizzle.config.ts` updated: change `out` to a temp/ignored path (e.g. `.drizzle-tmp/`) so accidental `drizzle-kit generate` calls don't recreate the deleted `drizzle/` folder. Add `.drizzle-tmp/` to `.gitignore`.

**New workflow for schema changes:**
1. Edit Drizzle schema file (e.g. add column in `schema/devices.ts`)
2. Write migration: `apps/api/migrations/0066-add-foo-to-devices.sql`
3. Run `pnpm db:check-drift` — should report no drift
4. If drift detected, the migration is incomplete

## Risks

| Risk | Mitigation |
|------|------------|
| Baseline doesn't match existing DBs | Validate by diffing pg_dump of baseline vs old system |
| Legacy detection misidentifies DB state | Simple check: `users` exists + `breeze_migrations` doesn't = legacy |
| Existing deployers confused by migration path change | Document in release notes, test upgrade path |
| Checksum mismatch on relocated manual migrations | Preserve file content exactly, only change filename |
| Accidental `drizzle-kit generate` recreates deleted folder | Redirect `out` in `drizzle.config.ts` to `.drizzle-tmp/`, gitignored |
| `seed()` depends on global Drizzle `db` instance | Preserve initialization order: migrations first, then seed via global client |

## Success Criteria

1. Fresh database: `docker compose up` starts API successfully, all tables/indexes/RLS created
2. Existing database (SemoTech's state): API starts, no migration errors, schema unchanged
3. Legacy `db:push`-created database: API detects legacy state, baselines correctly, starts (one-time migration test for existing deployments)
4. `pnpm db:check-drift` reports no drift after applying all migrations
5. No `drizzle-kit generate` in any CI or developer workflow
6. CI passes with updated drift check step
